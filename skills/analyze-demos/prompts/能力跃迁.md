# 模型能力跃迁分析提示词

## 任务

分析这个 Demo 场景下，**豆包1.6为什么做不到或做不好**，以及**豆包1.8因为什么新能力可以做到**。

---

## 输入

- **Prompt描述**：Demo 的功能描述
- **模型能力**：已识别的能力标签（1-3个）

---

## 能力差异知识库

### 豆包 1.6 → 1.8 核心升级

| 维度 | 1.6 局限 | 1.8 突破 |
|------|----------|----------|
| **多模态架构** | LLM + VLM 组合方案 | 原生多模态端到端训练 |
| **视频理解** | 640帧/次（约21秒@30fps） | 1280帧/次（翻倍）+ 关键帧高精度 |
| **Agent能力** | GUI操作（浏览器级） | OS Agent（电脑/网页/移动端） |
| **工具调用** | 基础工具调用 | 思考+工具一体化，状态保留 |
| **指令遵循** | 支持 | 大幅增强，多轮多步更强 |
| **代码生成** | 可生成 | 更稳定，长代码格式更规范 |

### 能力标签与升级对应关系

| 能力标签 | 1.6 具体局限 | 1.8 具体突破 |
|----------|--------------|--------------|
| **多模态输入** | 640帧视频上限，超长视频需分段处理 | 1280帧 + 低帧率全局理解 + 关键帧高精度分析 |
| **代码狂魔** | 长代码偶有格式问题，复杂项目稳定性不足 | 工具调用稳定性极高，复杂代码输出规范 |
| **3D空间感** | 基础空间理解 | 空间理解评测大幅提升 |
| **物理引擎脑** | 基础物理模拟理解 | 物理规律理解更精准 |
| **Agent自动化** | 仅浏览器GUI操作，需页面可见 | OS级操作，跨应用自动化，无需API |
| **超长待机** | 256K上下文，管理能力有限 | 更灵活的上下文管理，复杂任务不丢失 |
| **工具达人** | 基础工具调用，多步易断链 | 思考+工具一体化，多步工具链稳定 |
| **指令遵循王** | 支持复杂指令 | 大幅增强，多轮交互指令遵循更精准 |
| **深度推理** | 4档思考模式 | 思考状态保留，推理链更完整 |
| **图像生成** | 支持 | 无显著变化，可描述为"延续强大能力" |

---

## 输出格式

使用以下结构（60-120字）：

```
1.6受限于[具体局限点]，[无法/难以][实现什么效果]；1.8凭借[新能力名称]，[可以/能够][实现什么效果]。
```

**格式要点**：
- 以"1.6受限于"开头
- 中间用分号"；"分隔
- 后半句以"1.8凭借"开头
- 局限和突破要**具体对应**，不要泛泛而谈

---

## 示例

### 示例1
**输入能力**：["多模态输入", "超长待机"]
**输入描述**：2小时网课视频自动生成笔记

**输出**：
1.6受限于640帧视频理解上限，无法完整理解2小时超长视频，需要分段处理导致上下文断裂；1.8凭借1280帧视频理解能力和低帧率全局+关键帧高精度分析技术，可以一次性理解完整课程并精准提取核心知识点。

### 示例2
**输入能力**：["Agent自动化", "工具达人"]
**输入描述**：自动整理收件箱邮件并分类归档

**输出**：
1.6受限于浏览器级GUI操作，无法跨应用操作邮件客户端和文件系统；1.8凭借OS Agent能力，可以直接操作桌面邮件应用、读取附件、创建文件夹，实现全流程自动化归档。

### 示例3
**输入能力**：["代码狂魔", "指令遵循王"]
**输入描述**：一句话生成完整的3D乐高编辑器

**输出**：
1.6受限于长代码生成稳定性不足，复杂项目容易出现格式错误或功能遗漏；1.8凭借增强的指令遵循能力和更稳定的代码输出，可以一次性生成结构完整、功能齐全的3D编辑器应用。

### 示例4
**输入能力**：["3D空间感", "物理引擎脑"]
**输入描述**：台球物理引擎模拟游戏

**输出**：
1.6受限于基础空间理解能力，生成的物理模拟参数不够精准，球体碰撞和运动轨迹偏差较大；1.8凭借提升的空间理解和物理规律理解能力，可以生成更真实的碰撞反弹效果和精准的运动轨迹。

### 示例5
**输入能力**：["多模态输入"]
**输入描述**：手写食谱识别并整理成电子版

**输出**：
1.6受限于组合式多模态架构，对复杂手写体和不规范排版的识别准确率有限；1.8凭借原生多模态端到端训练，对潦草字迹和非标准格式的理解能力显著提升，识别准确率更高。

---

## 注意事项

1. **能力对应**：根据输入的能力标签选择对应的局限/突破描述
2. **具体化**：使用具体数字（640帧→1280帧）或具体场景，避免空泛
3. **因果关系**：局限导致"无法/难以"，突破实现"可以/能够"
4. **不要编造**：严格基于知识库中的能力差异，不要虚构不存在的能力
5. **多能力组合**：如果有多个能力标签，选择最核心的1-2个进行对比

---

## 特殊情况处理

### 如果能力标签是"图像生成"
由于1.6和1.8在图像生成能力上无显著差异，可以这样描述：
"1.6已具备图像生成能力，但在复杂场景理解上存在局限；1.8延续强大的图像生成能力，并凭借原生多模态架构，对复杂场景的理解更加准确，生成质量更稳定。"

### 如果无法明确对应能力
使用通用的架构升级描述：
"1.6采用组合式架构，在复杂任务中存在协调损耗；1.8凭借原生端到端训练，各项能力融合更紧密，执行效率和质量全面提升。"
